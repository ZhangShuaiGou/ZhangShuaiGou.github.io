Count Paths on a Game Board

Today, I practiced an interesting problem in Firecode.com.

The description of the question is:

You’re given a game board that has m x n squares on it, represented by an m x n array. Write a method - count_paths that takes in m and n and returns the number of possible paths from the top left corner to the bottom right corner. Only down and right directions of movement are permitted.

In other words, given a matrix [m,n] and output the distinct paths from [0,0] to [m,n].

At the beginning, I was thinking about using a tree to represent all the decisions - because only down and right directions are permitted. I noticed an interesting thing, that is once we need to reach to [m,n], we have to approach [m,n-1] and [m-1,n] and the number of paths reaching [m,n] equals the paths reaching [m,n-1] + [m-1,n]. In conclusion, the time complexity of the first solution is O(2m+n). Because for every single time, the function count_paths will run twice. There are m + n times in total. However, we can use some strategy to prune the algorithm. It doesn’t improve too much. Especially when the m and n grows larger and larger.

However, the time complexity of the second solution is O(m\n)*. Because there are only two for() loops in the solution.

Apparently, the O(2m+n) will be extremely larger than O(m\n)* when the m and n grows larger.
